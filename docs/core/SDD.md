## 2. Architecture Overview

```
User action (sign up / update / delete)
    │
    ▼
Clerk detects the change
    │  fires signed HTTP POST via Svix
    ▼
ngrok public URL  (https://xxxx.ngrok.io/webhooks/clerk)
    │  tunnels to localhost
    ▼
POST /webhooks/clerk  (Fastify, port 3001)
    │  verify Svix signature → insert into SQLite
    ▼
webhook_events (SQLite)
    ▲
    │  GET /api/events (polled every 3s)
    │
React frontend (port 5173)
    └── TanStack Query → EventsPage → EventList + PayloadInspector
```

**Svix** is Clerk's internal webhook delivery engine. It signs every request and retries on non-2xx responses. You do not interact with Svix directly.

**ngrok** exposes your local backend to the internet so Svix can reach it. Register the ngrok URL in the Clerk Dashboard under Webhooks.

## 3. Package: `packages/types`

Single source of truth for the `WebhookEvent` shape. Both the backend and frontend import from here — no type duplication, no drift.

```ts
// packages/types/src/index.ts
import { z } from 'zod'

export const WebhookEventSchema = z.object({
  id: z.string().uuid(),
  svix_id: z.string(),
  event_type: z.enum(['user.created', 'user.updated', 'user.deleted']),
  payload: z.record(z.unknown()),
  received_at: z.number(),
})

export type WebhookEvent = z.infer<typeof WebhookEventSchema>
```

**Backend usage:** validates rows before returning them from `GET /api/events`.

**Frontend usage:** types the result of the `useEvents()` TanStack Query hook.

## 4. Backend — `apps/backend`

### 4.1 Tech stack

| Concern | Library |
|---|---|
| HTTP server | Fastify |
| Webhook verification | `@clerk/backend` (`verifyWebhook`) |
| Database | `better-sqlite3` (synchronous SQLite driver) |
| Validation | Zod (via `packages/types`) |
| Runtime | Node.js |

### 4.2 File structure

```
apps/backend/
├── src/
│   ├── index.ts            ← Fastify app setup, plugin registration, server start
│   ├── routes/
│   │   ├── webhooks.ts     ← POST /webhooks/clerk
│   │   └── events.ts       ← GET /api/events, DELETE /api/events
│   └── db/
│       ├── index.ts        ← SQLite connection, table initialisation on startup
│       └── events.ts       ← insertEvent(), findAll(), deleteAll()
├── .env.example
└── package.json
```

### 4.3 Database schema

```sql
CREATE TABLE IF NOT EXISTS webhook_events (
  id          TEXT    PRIMARY KEY,   -- UUID v4, generated by backend on insert
  svix_id     TEXT    NOT NULL UNIQUE, -- Svix message ID, used for idempotency
  event_type  TEXT    NOT NULL,       -- 'user.created' | 'user.updated' | 'user.deleted'
  payload     TEXT    NOT NULL,       -- full raw JSON string from Clerk
  received_at INTEGER NOT NULL        -- Unix timestamp in milliseconds
);
```

**Design notes:**

- `id` is `TEXT PRIMARY KEY`. Generated via `crypto.randomUUID()` on each insert. No autoincrement — UUIDs are safe to expose in API responses without leaking record counts.
- `svix_id` is `UNIQUE`. Svix retries delivery on any non-2xx response. This constraint makes inserts idempotent: a duplicate `svix_id` causes the insert to be silently skipped, and the endpoint still returns `200 OK`.
- `payload` is stored as a raw JSON string. Event types have different shapes; storing the full blob preserves the original for display in the inspector and avoids designing a column schema per event type.
- No `verified` column. Signature verification is enforced at the route level before any DB write. Every row in the table is implicitly verified.

### 4.4 Routes

#### `POST /webhooks/clerk`

```
Headers required:
  svix-id, svix-timestamp, svix-signature   (added by Svix automatically)

Body:
  Raw bytes — must NOT be parsed as JSON before verification

Flow:
  1. Read raw body as string
  2. Call verifyWebhook(body, headers, secret) from @clerk/backend
     → throws on invalid signature → return 400
  3. Extract event type and svix-id from verified payload
  4. INSERT OR IGNORE into webhook_events (idempotent)
  5. Return 200 OK
```

**Important:** Fastify must be configured with `addContentTypeParser` for `application/json` to return the raw body as a string on this route. Parsing the body before verification corrupts the signature check.

#### `GET /api/events`

```
Response: WebhookEvent[]   (last 50, ordered by received_at DESC)
  payload field is parsed back from JSON string to object before returning
```

#### `DELETE /api/events`

```
Truncates webhook_events table.
Returns 204 No Content.
No auth — local dev only.
```

### 4.5 Error handling

| Scenario | Response | Reason |
|---|---|---|
| Invalid Svix signature | `400 Bad Request` | Reject before DB write. Teaches why verification matters. |
| Duplicate `svix_id` | `200 OK`, insert skipped | Svix retries on non-2xx. Returning 200 signals successful receipt. |
| Malformed request body | `400 Bad Request` | Handled automatically by Fastify. |

### 4.6 CORS

The backend must allow requests from `http://localhost:5173` (the Vite dev server). Use `@fastify/cors`.

### 4.7 Environment variables

```bash
# apps/backend/.env.example
CLERK_WEBHOOK_SIGNING_SECRET=whsec_...   # Clerk Dashboard → Webhooks → your endpoint → Signing Secret
PORT=3001
```

## 5. Frontend — `apps/frontend`

### 5.1 Tech stack

| Concern | Library |
|---|---|
| Framework | React + Vite |
| Routing | TanStack Router |
| Data fetching | TanStack Query |
| Auth | Clerk React SDK |
| Syntax highlighting | Shiki |
| Types | `packages/types` |

### 5.2 File structure

```
apps/frontend/src/
├── routes/
│   ├── __root.tsx          ← RootLayout: ClerkProvider, QueryClientProvider, RouterProvider
│   └── index.tsx           ← single page: AuthPanel + EventInspector side by side
├── components/
│   ├── AuthPanel.tsx        ← renders <SignIn /> or <UserProfile /> based on useUser()
│   ├── EventInspector.tsx   ← splits into EventList + PayloadInspector, manages selected state
│   ├── EventList.tsx        ← renders list of EventListItem rows
│   ├── EventListItem.tsx    ← single row: type badge, email, relative timestamp
│   └── PayloadInspector.tsx ← Shiki-highlighted JSON of selected event's payload
├── api/
│   └── events.ts           ← fetchEvents() function + useEvents() TanStack Query hook
├── lib/
│   └── shiki.ts            ← module-level Shiki singleton, exports highlight()
└── main.tsx
```

### 5.3 Route tree

Only one real route. TanStack Router is used for its type-safe link/navigation utilities, not for multi-page navigation.

```
__root  (RootLayout)
  └── /  (index — the entire app)
```

### 5.4 Auth panel behaviour

```
useUser() from Clerk
  │
  ├── isLoaded === false  →  loading spinner
  ├── isSignedIn === false  →  <SignIn />   (has built-in "Sign up" link)
  └── isSignedIn === true   →  <UserProfile />  (profile update + account deletion)
```

`<SignIn />` and `<UserProfile />` are Clerk's pre-built components. No custom auth forms needed.

### 5.5 Data fetching

```ts
// src/api/events.ts
export function useEvents() {
  return useQuery({
    queryKey: ['events'],
    queryFn: fetchEvents,
    refetchInterval: 3000,   // poll every 3 seconds
  })
}
```

`fetchEvents` calls `GET /api/events` and validates the response against `WebhookEventSchema` array from `packages/types`.

Selected event state is managed locally in `EventInspector` with `useState`. No global state needed — the full payload is already in the query result.

### 5.6 Payload highlighting (Shiki)

Shiki is initialised once as a module-level singleton in `src/lib/shiki.ts`, exporting a `highlight(code: string): Promise<string>` function. `PayloadInspector` calls it with `JSON.stringify(payload, null, 2)` and renders the returned HTML string with `dangerouslySetInnerHTML`. Initialising at the module level avoids prop-drilling or context setup, and ensures the highlighter is only created once regardless of how many times the component mounts.

## 7. Local Development Setup

1. Install dependencies: `pnpm install` from monorepo root
2. Copy `.env.example` to `.env` in both `apps/backend` and `apps/frontend`, fill in values from Clerk Dashboard
3. Start backend: `pnpm --filter backend dev` (port 3001)
4. Start frontend: `pnpm --filter frontend dev` (port 5173)
5. Start ngrok: `ngrok http 3001`
6. Register the ngrok HTTPS URL + `/webhooks/clerk` path in the Clerk Dashboard under **Webhooks**, subscribing to `user.created`, `user.updated`, `user.deleted`
7. Copy the **Signing Secret** from the webhook endpoint into `CLERK_WEBHOOK_SIGNING_SECRET` in `apps/backend/.env`
8. Open `http://localhost:5173` and sign up — watch the event appear in the inspector

## 8. Key Teaching Points

| Concept | Where it appears in the code |
|---|---|
| Webhook signature verification | `routes/webhooks.ts` — `verifyWebhook()` call |
| Why raw body matters | `routes/webhooks.ts` — content type parser config |
| Idempotent processing | `db/events.ts` — `INSERT OR IGNORE` + `UNIQUE` on `svix_id` |
| Svix retry behaviour | `db/events.ts` + error handling table |
| Shared types in a monorepo | `packages/types` imported by both apps |
| Polling vs real-time | `api/events.ts` — `refetchInterval: 3000` |
| Cause and effect | Single-page layout — action on left, event on right |
